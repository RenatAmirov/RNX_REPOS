Pour cette première étape, nous allons générer le squelette de notre projet. Le plus simple pour cela est d’utiliser Angular CLI.

Installation d’Angular CLI

Note: Si vous avez déjà une version de Angular CLI, vous devez d’abord la désinstaller avec npm uninstall -g @angular/cli

Commençons donc par installer cet outil :

npm install -g @angular/cli@22.0.0

Cela ajoute un exécutable nommé ng à votre ligne de commande. Vous devriez pouvoir vérifier la version avec :

ng version

Cela devrait afficher la version spécifiée après le @ dans la commande d’installation. Sinon consultez la FAQ sur l’installation et la mise à jour de Angular CLI.

Génération du projet

Puis placez-vous dans un dossier fraîchement créé et entrez :

ng new ponyracer --defaults --no-routing --prefix pr --skip-install

pour générer un nouveau projet.

Après quelques secondes (voire minutes), vous devriez voir apparaître un nouveau dossier, nommé ponyracer (selon votre environnement vous aurez peut-être quelques warnings, rien de grave).

Remplacez ensuite le fichier package.json à la racine de votre application (soit directement en cliquant sur le fichier ci-dessus dans la section "Resources" ou depuis le zip "Setup" téléchargé contenant tous ces fichiers), pour ajouter quelques dépendances, notamment ng2ninja qui sera responsable d’analyser votre code et de soumettre vos résultats au site.

Lancez maintenant npm install dans le projet pour télécharger les dépendances :

npm install

NPM va peut-être vous proposer de lancer une commande d’audit à la fin de l’installation : même si c’est tentant, ne la lancez pas, car cela va récupérer des versions possiblement incompatibles avec notre projet.

Si l’installation échoue à cause d’une dépendance incompatible, exécutez la commande de configuration suivante :

npm config set legacy-peer-deps=true --location=project

Et essayez la commande npm install à nouveau.

Angular CLI génère un fichier de configuration Prettier par défaut (.prettierrc) lors de la création d’un nouveau projet. Remplacez-le par celui fourni dans les ressources pour utiliser nos règles de formatage personnalisées.

Depuis la CLI v12, il n’y a plus de tests de bout en bout (end-to-end ou e2e) par défaut. Nous devons choisir une bibliothèque de test e2e, et l’ajouter au projet.

Nous allons utiliser Playwright. On pourrait le configurer à la main, mais le plus simple est d’utiliser une "schematic". La CLI est un outil très puissant. Elle permet aux développeurs de bâtir leurs propres "extensions" de la CLI (appelées "builders"), et leurs propres générateurs de code (appelés "schematics").

Cette Playwright schematic est la schematic fournie par la communauté que nous allons utiliser pour ajouter Playwright à notre projet :

cd ponyracer
ng add playwright-ng-schematics@21.1.0 --install-browsers --skip-confirmation

Si l’installation échoue à cause des peer dependencies, exécutez :

npm config set legacy-peer-deps=true --location=project

Et essayez la commande ng add à nouveau.

Pendant longtemps, la CLI incluait également l’outil TSLint par défaut pour vérifier notre code. Mais TSLint est maintenant déprécié, il n’est donc plus inclus depuis la CLI v12. Pas d’inquiétude cependant : nous allons utiliser ESLint à la place.

ng add angular-eslint@21.4.0 --skip-confirmation

Parfait, tout est installé, on peut passer à la suite.

Démarrer l’application

Mettez maintenant à jour la configuration de la CLI en remplaçant le fichier angular.json file par celui fourni.

Tout est déjà configuré pour nous : l’outil de build, les dépendances, les tests, et même nos premiers composants !

Le projet est par défaut sous Git, et utilise Vite pour être construit.

Vous pouvez vous placer dans ce dossier ponyracer et démarrer le projet localement avec:

ng serve

Puis ouvrir votre navigateur à l’adresse http://localhost:4200

Vous devriez voir la page d’accueil s’afficher.

Un peu de style

Allons voir le code maintenant.

Ouvrez le projet avec votre IDE préféré. Visual Studio Code est par exemple gratuit, disponible pour tous les systèmes d’exploitation, et possède un très bon support de TypeScript. Tu peux lui ajouter le plugin Angular Language Service qui donne le support d’Angular, ainsi que le plugin ESLint.

Vous pouvez voir que notre application se situe dans le répertoire src.

Commençons par essayer de rendre cette application un peu plus jolie. Mes faibles compétences en CSS m’amène souvent à utiliser l’excellent Bootstrap.

Installons donc bootstrap et fontawesome :

npm install --save-exact bootstrap@5.3.8 @fortawesome/fontawesome-free@7.2.0

Nous devons maintenant utiliser ces styles dans notre application. Ouvrez le fichier styles.css et ajoutez les imports suivants :

styles.css
@import 'bootstrap/dist/css/bootstrap.css';
@import '@fortawesome/fontawesome-free/css/all.css';

Puis dans le template du composant app.html, remplaçons le contenu par :

app.html
<main class="container" style="margin-top: 70px;">
  <h1>Ponyracer</h1>
</main>

Relancez la tâche ng serve et votre navigateur devrait se rafraîchir automatiquement et afficher notre nouveau template.

La configuration TypeScript est dans le fichier tsconfig.json, et, comme nous l’avons vu, le compilateur va nous aider à écrire du bon code. Nous pouvons ajouter quelques options au compilateur pour le rendre un peu plus strict et l’aider à trouver d’autres erreurs potentielles dans notre code. Remplacez le fichier généré par la CLI par celui fourni dans les ressources.

Tests unitaires

Reste maintenant à ajouter de quoi valider nos exercices !

Copiez le fichier exercise.json à la racine de votre application qui permet à notre script de savoir quel est l’exercice en cours.

Configurons Vitest en ajoutant un fichier de configuration :

ng generate config vitest

Puis remplacez le fichier vitest-base.config.ts généré à la racine de votre application par celui fourni dans les ressources.

Copiez le fichier tsconfig.spec.json à la racine de votre application, et copiez également le fichier setup-test.ts dans le dossier src. Ce fichier configure l’environnement de test Vitest : il ajoute une vérification pour s’assurer qu’aucune erreur console ne se produit pendant les tests, et étend Vitest avec une méthode utilitaire.

Commençons par voir si les tests unitaires sont OK. Créez un dossier results à la racine de votre projet. Puis remplacez app.spec.ts dans votre projet (src/app) par celui fourni dans les ressources, et lancez :

ng test

Par défaut, les tests sont en mode "watch", ce qui signifie qu’ils vont se relancer à chaque modification de fichier. Pour exécuter les tests une seule fois, vous pouvez lancer :

ng test --no-watch

Vous pouvez aussi afficher une UI avec l’option --ui.

Les tests vérifieront aussi la couverture de code de votre application. Pour lancer les tests avec la couverture de code, utilisez l’option --coverage :

ng test --coverage --no-watch

Les résultats de couverture sont affichés dans le terminal (résumé textuel), et un rapport HTML est généré dans le dossier coverage/. Ouvrez coverage/ponyracer/index.html dans votre navigateur pour explorer le rapport détaillé.

Tests de bout en bout (e2e)

Ensuite, regardons si les tests end-to-end sont OK. Copiez home.e2e-spec.ts dans le répertoire e2e, et supprimez le test par défaut example.spec.ts. Playwright a son propre fichier de configuration, remplacez le fichier playwright.config.ts à la racine du projet par celui fourni. Ajouter également le fichier tsconfig.json dans le répertoire e2e fourni dans les ressources.

Ensuite, lancez les tests avec :

ng e2e
Note
 Sur certains systèmes, vous devrez peut-être exécuter npx playwright install-deps pour installer les dépendances système manquantes. Vous devriez avoir un message d’erreur explicite si c’est le cas.

Si vous voulez avoir une interface graphique pour voir les tests s’exécuter, vous pouvez lancer les tests avec :

ng e2e --ui

Lint

Dernière étape, vérifions la qualité de notre code. Les packages nécessaires sont déjà installés, il ne manque plus qu’à configurer l’outil. Pour cela, remplacez le fichier eslint.config.js à la racine du projet par le fichier eslint.config.mjs fourni (notez l’extension .mjs) et vérifiez la qualité du code avec :

ng lint

Corrigez toutes les erreurs (s’il y en a) pour atteindre 100%.

Notez que vous pouvez lancer :

ng lint --fix

à n’importe quel moment pour automatiquement formatter votre code, et vous débarrasser d’erreurs ennuyeuses comme des espaces ou points-virgules manquants.

Envoi des résultats

Tout est bon ? OK, soumettons notre résultat alors !

Pour lancer l’analyse du code et la soumission de votre résultat au site, entrez :

npm run ng2ninja

Vous devrez entrer votre email et mot de passe pour pouvoir soumettre votre résultat.

Si vous avez suivi les consignes, vous devriez obtenir votre premier 100% et pouvoir passer à l’étape suivante !
